# Billing Policy

우리 소싱 솔루션의 과금, 구독, 권한 확인 방식에 대한 기획 문서.

주의사항:

- 이 문서는 기획 기준 문서이며, 실제 코드 반영은 별도 요청 시 진행한다.
- 서비스는 `개인 구독형 SaaS` 구조를 기본으로 한다.
- 과금은 단순 결제 처리만이 아니라 `플랜 관리 + 구독 상태 + 권한(entitlement) 확인`까지 함께 설계한다.

## 기본 방향

- 과금 방식은 `월 구독제`를 기본으로 한다.
- 사용량 기반 과금보다 `권한 기반 접근 제어` 방식이 더 적합하다.
- 사용자는 기능 버튼 수보다 `데이터 접근 수준`에 대해 비용을 지불하는 구조로 설계한다.

한 줄 요약:

`결제 자체보다 구독 상태와 권한(entitlement)을 기준으로 서비스 접근을 제어한다.`

## 추천 플랜 구조

초기 버전은 단순한 플랜 구조로 시작한다.

추천 예시:

- `Free Trial`
- `Basic`
- `Pro`

### Free Trial

- 7일 또는 14일 체험
- 일부 테마만 열람 가능
- 저장 개수 제한
- 상세 데이터 일부 제한

### Basic

- 전체 추천 상품 열람
- 기본 필터와 저장 기능 제공
- 일 단위 갱신 데이터 접근 가능

### Pro

- 전체 카테고리 열람
- 고급 지표 열람
- 고급 필터 사용 가능
- 추후 쿠팡 보강 데이터 열람 가능

## 과금 제한 기준

이 서비스는 API 호출 수 자체보다 `어떤 데이터와 기능에 접근 가능한가`를 기준으로 제한하는 것이 적합하다.

추천 제한 기준:

- 열람 가능한 테마 수
- 하루 확인 가능한 추천 개수
- 상세 데이터 노출 깊이
- 저장 가능한 상품 수
- 고급 필터 사용 여부
- 급상승/저경쟁/고마진 지표 사용 여부
- 추후 쿠팡 확장 데이터 열람 여부

## 결제 수단 방향

국내 중심 서비스 기준 권장 순서:

1. `토스페이먼츠`
2. `아임포트(PG 연동)`
3. `Stripe` (해외 확장 시)

초기 버전에서는 국내 대응이 쉬운 `토스페이먼츠`가 가장 현실적이다.

## 확인 방식의 핵심 원칙

결제 성공 여부만으로 접근을 판단하지 않는다.

반드시 서버 내부의 구독 상태와 권한 값을 기준으로 접근을 제어한다.

기본 흐름:

1. 결제 성공
2. PG webhook 수신
3. 서버가 구독 상태 갱신
4. 사용자 요청 시 현재 구독 상태와 entitlement 확인
5. 허용 범위 내 기능만 노출 및 실행

즉, 프론트에서 결제 완료 화면을 봤다고 바로 권한을 주는 구조가 아니라, 서버가 최종 권한 상태를 관리해야 한다.

## 권장 데이터 구조

### `plans`

- id
- name
- price
- billing_cycle
- is_active

### `subscriptions`

- id
- user_id
- plan_id
- status
- started_at
- current_period_end
- canceled_at
- grace_until

### `payments`

- id
- user_id
- subscription_id
- provider
- provider_payment_id
- amount
- status
- paid_at

### `entitlements`

- id
- plan_id
- code
- value

예시 entitlement:

- `max_devices = 2`
- `max_saved_items = 100`
- `can_view_coupang_data = false`
- `can_use_advanced_filters = true`

## 구독 상태 설계

권장 상태값:

- `trialing`
- `active`
- `past_due`
- `canceled`
- `expired`

설명:

- `trialing`: 무료 체험 중
- `active`: 정상 구독 상태
- `past_due`: 갱신 실패, 유예기간 상태
- `canceled`: 해지 처리됨 또는 해지 예약 상태
- `expired`: 접근 제한 상태

## 결제 실패 처리

결제 실패 시 즉시 차단하지 않는다.

권장 정책:

- `3일~7일` grace period 운영
- 이 기간에는 서비스 유지 또는 일부 기능 유지
- grace 기간 내 재결제 유도
- grace 기간 종료 시 유료 기능 차단
- 기존 저장 데이터는 삭제하지 않는다

예시:

- 결제 실패 1일차: 알림 발송
- 결제 실패 3일차: 재결제 안내 강화
- 결제 실패 7일차: 플랜 만료 처리

## 사용자 화면에서 필요한 정보

PWA 사용자 화면에는 최소한 다음 항목이 필요하다.

- 현재 플랜
- 다음 결제일
- 결제 상태
- 남은 체험 기간
- 결제수단 변경
- 구독 해지
- 결제 내역/영수증
- 현재 등록 기기 수

추천 메뉴명:

- `내 계정`
- `구독 및 결제`

## 관리자 화면에서 필요한 정보

관리자 화면에는 다음 항목이 필요하다.

- 사용자별 플랜 상태
- 결제 성공/실패 내역
- 무료 체험 사용자 목록
- 해지 예정 사용자 목록
- 환불 처리 메모
- 강제 플랜 변경
- 프로모션 적용 여부

## API 권한 확인 흐름

서버는 주요 요청마다 다음 순서로 확인한다.

1. 로그인 사용자 확인
2. 계정 상태 확인
3. 구독 상태 확인
4. 해당 기능에 필요한 entitlement 확인
5. 허용되면 데이터 반환
6. 허용되지 않으면 접근 차단 응답 반환

예시:

- 추천 목록 조회: `trialing` 또는 `active` 허용
- 고급 필터: 특정 entitlement 필요
- 쿠팡 확장 데이터: `Pro` 플랜만 허용
- 저장 개수 초과 시 추가 저장 차단

## 초기 버전 권장안

V1에서는 복잡한 과금 구조보다 단순한 구독 검증이 중요하다.

추천 초기안:

- 무료체험 7일
- 월 구독 1종 또는 2종
- 계정당 기기 2대 허용
- 결제 실패 시 3일~7일 유예
- 서버 기준 entitlement 확인
- PG webhook 기반 상태 동기화

초기에는 다음 항목은 나중으로 미뤄도 된다.

- 연간 요금제
- 팀 플랜
- 좌석제
- 복잡한 쿠폰 시스템
- 다중 통화 결제

## 결론

최종 과금 방향은 다음과 같다.

- 월 구독제를 기본으로 한다.
- 결제 성공 여부가 아니라 구독 상태를 기준으로 권한을 판단한다.
- 기능 제한보다 데이터 접근 수준 중심으로 과금 차별화를 둔다.
- entitlement 구조를 통해 플랜별 정책을 유연하게 관리한다.
- 결제 실패 시 즉시 차단하지 않고 유예기간을 둔다.

한 줄 요약:

`과금은 월 구독제로 가고, 확인은 결제 자체가 아니라 서버의 구독 상태와 권한(entitlement)을 기준으로 처리한다.`
